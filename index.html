发表于 2019-02-25 | 分类于 性能

加载和运行

  • 浏览器以单线程处理ui和js任务，处理方式为串行，一次执行一个任务。（如果浏览器为多线程，浏览器整个设计复杂程度会提高很多。）
  • 浏览器在碰到script标签时，停止其他任务，然后下载script，解析，执行里面的代码。script标签具有阻塞作用。（浏览器遇到href不会阻塞页面，会并行下载，执行其中代码，href为在页面和目标文件之间一种联系，而src则是指向一个资源，相当于插入一段资源，会阻塞页面。所以一般将script标签放在前一行。link标签放在head中，优先加载基本样式。当然script标签放在head中䦹可以的，可以添加async标签，进行异步加载。chrome稳定版中，同一域名下并发请求数为6，chrome相关源码）
  • 下载一个100k的script比四个25k的script要快，所以尽可能将小的script合并在一个文件中。
  • 利用cdn网络，加速下载。（内容分发网络。CDN是构建在网络之上的内容分发网络，依靠部署在各地的边缘服务器，通过中心平台的负载均衡、内容分发、调度等功能模块，使用户就近获取所需内容，降低网络拥塞，提高用户访问响应速度和命中率。）
  • 利用script的defer，延迟加载属性，在页面dom完成之后，onload方法调用之前加载脚本。
  • 多数浏览器在下载动态script时，并不会按照顺序下载。
  • 利用XHR下载脚本，并动态插入到页面中.xhr请求涉及到跨域问题,只能请求不跨域的资源.

数据访问

  • 数据存储在哪里，关系到代码运行期间数据被检索到的速度，在javascript中有四种基本的数据访问的位置（1直接量，2变量，3数组项，4对象成员）。
  • javascript内部属性[[Scope]]包含一个函数被创建的作用域中对象的集合。此集合被称为函数的作用域链，它决定哪些数据可由函数访问。此函数作用域链中的每个对象被称为一个可变对象，每个可变对象都以“键值对”的形式存在。
  • 运行一个函数时，就会建立一个内部对象，称为“运行期上下文”。一个运行期上下文定义了一个函数运行时的环境。它是独一无二的。多次调用函数，会多次创建运行期上下文。函数执行完毕，运行期上下文就销毁。
  • 创建函数时，会填入一个单独的全局对象到被创建的作用域链(用于标识符解析)中，此全局对象包括访问this对象，window对象，document对象，本身的接口。同时也是创建一个被称为“激活对象”填入到作用域链中，此对象包括可访问this，函数参数，函数内部局部变量的接口。
  • 函数运行中，每遇到变量，标识符识别过程就要决定从哪里获取或存储数据。就会在作用域链中查找相同的标识符，从作用域链的最前端开始，没找到，就往下一个对象查找，直到找到为止。一个标识符的所处的位置越深，读写速度就越慢，全局对象位置最深，出于作用域链中的最后一个位置，通常来说速度最慢。局部变量速度最快。而采用优化的js引擎的浏览器没有此问题。
  • 在局部作用域中，如果全局变量使用次数多于一次，可以使用局部变量保存全局变量。再调用局部变量。

  • 可以改变运行期上下文的作用域链的方法：with和try catch。

  • 对象成员中，多次访问的属性，可以用局部变量保存。避免过深的对象属性嵌套

dom编程

  • 避免多次读写dom对象
  • 在使用传统循环时，可以将length用变量保存，避免每次都需要查询length。
  • 使用 querySelectorAll比其他获取dom的api更快，且参数类似css选择器。
  • 触发回流的行为:1添加或删除可见的DOM元素 2元素位置改变 3元素尺寸改变（边距，填充，边框宽度，高度等属性改变） 4内容改变（被更大或更小的尺寸替代） 5最初的页面渲染 6浏览器窗口尺寸改变
  • 触发回流的原生api：offsetTop offsetLeft offsetWidth offsetHeight scrollTop scrollLeft scrollWidth scrollHeight clientTop clientLeft clientWidth clientHeight getComputedStyle
  • 使用cssText 合并dom修改，使用预先定义的className修改dom
  • 对一个dom元素要进行多次修改时，进行一下操作
    a 从文档中去除该元素（display none）/ 使用createDocumentFragment(因为文档片段存在于内存中，并不在DOM树中，所以将子元素插入到文档片段时不会引起页面回流（对元素位置和几何上的计算）。因此，使用文档片段通常会带来更好的性能。过多的插入元素，会使得内存暴涨，同样影响性能。) / 拷贝元素，应用修改，覆盖原始元素
    b 对其应用改变
    c 从文档中恢复该元素
  • 避免进行页面大部分区域的回流，使用绝对定位的元素，使其脱离文档流。不影响真个页面。避免推移整个页面。
  • 每个子元素调用事件api，访问和修改子元素dom浪费了内存和性能，利用代理技术，由子元素的父元素代理子元素处理事件api。（利用事件api的内部对象event）

算法和流程控制

代码量少不一定运行速度快，代码量多不一定运行速度慢。

  • es3规定了四种循环（for循环，while循环，do-while循环，for-in循环）。
  • 倒序循环时性能优化中最常用的手段，能微微提高性能,较少循环迭代次数，能大大的提高循环性能。
  • 达夫设备循环法。
  • 使用if-else和switch时，条件数量大，倾向于使用switch，这样会使代码逻辑更加清晰，易于阅读。使用嵌套的if-else比单独使用if-else性能更加好（单独的if-else，每条都需要计算，增加了计算量）if-else更加倾向于连续的数据，而switch倾向于离散的数据。
  • 在大量数据需要判断时，查表法比if-else，switch性能更加好。查表法需要去除所有判断条件，操作转换成数组或者对象。Code：
var results = [result0, result1, result2, result3, result4, result5, result6, result7, result8, result9, result10] 
return results[value];

  • 限制定时器的数量，避免在一定时间内，出现大量定时器，最好控制只有一个定时器对象，意思是创建第二个定时器时，应该销毁前一个。

单体模式

单体模式顾名思义就是一个类中，它只有一个实例，意思就是第二次，第三次，n次创建的实例要与第一次创建的实例完全相同。
我们想要的效果：

var uni=new Universe();
var uni2=new Universe();
uni===uni2; //结果为true

要想得到这样的结果，我们应该怎么改造Universe的内部呢？
必须想办法保存第一次实例，然后后面实例化的时候，返回第一次保存的实例。所以有下列几个方法：

  • 用全局变量保存第一次实例。但是全局变量容易被污染。会造成原先保存的实例的消失。
  • 可以用构造函数的静态属性保存第一次实例，但是呢，静态属性也是暴露的，所以很容易被修改。
  • 可以使用闭包技术，外界不能操作内部，这样就能很好的保存第一次实例对象，代价是带来了闭包的额外开销。

静态属性中的实例

Code：

function Universe(){
    if(typeof Universe.instance === "object"){
        return Universe.instance;
    }

    this.start_time=0;
    this.bang="Big";

    Universe.instance=this;
};

var uni=new Universe();
var uni2=new Universe();
uni===uni2; //结果为true;

此模式中最大缺点就是静态属性对外面暴露，很容易被修改。

闭包中的实例

Code：

function Universe(){
    var instance=this;

    this.start_time=0;
    this.bang="Big";

    Universe=function(){
        return instance;
    };
}
var uni=new Universe();
var uni2=new Universe();
uni === uni2; //结果true

第一次创建实例，保存第一次实例对象，然后通过闭包重写了构造函数，返回第一次实例对象。

传统与现代继承模式的比较

“传统”这个术语只是单词”class(类)”的一种表现形式。“现代”模式表述为：其他任何不需要以类的方式考虑的模式。

使用类式继承时的预期结果

实现类式继承的目标是通过构造函数Child()获取来自另外一个构造函数Parent()的属性，从而创建对象。

//父类
function Parent(name){
    this.name=name || "Nei";
};

// 添加功能
Parent.prototype.say=function(){
    return this.name;
};

//子类
function Child(){};

inherit(Child,Parent);

inherit函数非原生api，需要自行实现。现在我们来看看怎么实现它。

类式继承模式#1——默认模式

一种默认模式就是使用Parent构造函数创建一个对象，然后将该对象赋值给子类Child的原型。下面是inherit函数第一种实现：

function inherit(C,P){
    C.prototype=new P();
};

牢记一点，原型属性应该指向一个对象，而不是函数。所以它必须指向一个有父构造函数所创建的对象，而不是父构造函数本身。换句话说，要使用new操作符来创建的对象。

追溯原型链

使用这种模式时，同时继承了自身的属性（name），以及原型链上的属性和方法。那原型链是怎么样工作的呢？当new Parent()时，会创建一个块，保存这个对象，同时保存了name属性。但是当我们调用say方法时，say方法并不在这个块中，但是通过proto属性（指向Parent的prototype属性），我们找到了say方法。

当我们使用了inherit函数之后，又发生了什么呢？如图：

图中可以看到Child构造函数内部什么都没有，只有Parent实例的对象属性和方法被指向到Child的prototype上，Child的prototype上本身是什么都没有。当new Child()时，所创建的对象的proto指向Child类的prototype。所以当调用say方法时，在#3上并没有找到该方法，它会向#2上去寻找，在#2上也没有找到该方法，但是因为new Parent()的proto指向Parent的prototype，所以我们能在#1上找到该方法。

还有一个解释this指向的问题。如图：

var kid=new Child();
kid.name="De";
kid.say(); //输出 De


在kid对象上生成了name属性，调用say时，此时的this指向kid。

此模式缺点

在继承时，我们将自身属性和原型链上的属性和方法都继承了，但是有时并不需要这些自身属性。在实例化时，并不支持子构造函数传参。Code：

var s=new Child('Seth');
s.say(); // 输出 "Nei";

类式继承模式#2——借用构造函数

此模式解决了默认模式自构造函数不能传参的问题。通过一些方法借用父类构造函数，传递子对象以绑定到this，并且转发任意的参数。Code：

function Child(a,c,b){
    Parent.apply(this,arguments);
};

当时这种方法只能继承Parent自身的属性和方法，不能继承原型链上的属性和方法。

//父构造函数
function Article(){
    this.tags=['js','css'];
};
var article=new Article();

function BlogPost(){};
BlogPost.prototype=article; //默认模式继承
var blog=new BlogPost();

function StaticPage(){
    Article.call(this);
};
var page=new StaticPage();

console.log(article.hasOwnProperty('tags')); //true
console.log(blog.hasOwnProperty('tags')); //false
console.log(page.hasOwnProperty('tags')); //true

2返回了false，因为tags属性并没有在blog对象上，而是在原型链上，而page对象上的tags是借用了构造函数时，获取父类的副本，而不是引用。

当我们修改tags，看看差异：

blog.tags.push('html');
page.tags.push('php');
console.log(article.tags.join(','));输出 js,css,html

因为blog和article的tags指向了同一个数组，而page的targs时重新创建的的一个副本。

原型链

当使用本模式时，原型链时如何工作的呢？Code：

function Parent(name){
    this.name=name | "Adam";
};

Parent.prototype.say=function(){
    return this.name;
};

function Child(name){
    Parent.apply(this,arguments);
};

var kid=new Child('Nei');
kid.name; //输出 Nei
typeof kid.say //输出 undefined

最后为什么会输出undefined呢？因为借用继承只会继承父类的自身属性和方法。并没有继承到父类的prototype属性和方法。所以也就找不到say方法。

通过借用构造函数实现多重继承

借用多个构造函数实现多重继承。Code：

function Cat(){
    this.legs=5;
    this.say=function(){
        return 'miao'
    };
};

function Bird(){
    this.wings=3;
    this.fly=function(){
        return true;
    };
};

function Child(){
    Cat.apply(this);
    Bird.apply(this);
}

优缺点

缺点：

  • 无法继承父类原型链上的属性和方法，并且原型也仅是添加可重用方法以及属性的位置，它并不会为每个实例重新创建原型。
    优点：
  • 可以获得父类的自身成员的真实副本，也不会存在子对象意外覆盖父对象舒心的风险。

类式继承模式#3——借用和设置原型

顾名思义就是先借用父类构造函数，然后再设置原型。Code：

function Child(a,b,c,d){
    Parent.apply(this,arguments);
};
Child.prototype=new Parent();

此模式优点：

  • 继承了父类的自身成员和原型链上的成员。

缺点：

  • 自身成员被继承了两次。
  • constructor紊乱。子类构造函数应该指向本身，但是却指向了父类构造函数。

类式继承模式#4——共享原型

Code:

C.prototype=P.prototype;

Orinoco垃圾回收器[译]

发表于 2019-01-30

在过去的许多年，v8的垃圾回收器发生了很大的变化。这次，Orinoco项目采用了一个连续，能够并行且并发和回退功能的异常强大的收集器。

垃圾收集器会周期性的执行下列几个基本的任务：

  • 鉴别正在运行或者已经没被调用的对象
  • 回收/再利用已经没被调用的对象的内存
  • 清理内存

这些任务一般按顺序执行，有时也会交错执行。在主线程上一些直截了当的方法暂停js执行和按顺序执行所有的任务，这样会造成不可避免和潜在的问题，我们在之前的博客谈论过.

主要GC

GC从堆中收集垃圾

全局GC发生在三个阶段：标记，清楚，整理

标记

垃圾收集器必不可少的就是找出哪些对象能被收集。收集器使用可达性(reachability)代理活跃度(liveness).这意味着在运行时当前一些可访问(reachable)的对象将保留，而那些不可访问对象将被收集起来。

标记就是可访问对象被找到的过程，GC就是在一系列一直被称为根集(root set)的对象指针开始的。这其中包括执行栈和全局对象。遵循一个指针对应一个对象，然后将这个对象标记为可访问的。GC持续递归的跟踪这些对象的指针，直到每个可访问对象被找到和标记。

清除

清除就是非活动对象被添加到空闲列表，所处的那块内存被清理的过程。一旦标记完成，GC会找到不可访问对象的那块连续内存，然后将它们添加到适当的空闲列表里。所谓的空闲列表就是由可以快速查找的内存块分出来的。在未来，当我们想要分配内存时，我们仅仅只需要考虑空闲列表，并且找到一块大小合适的内存块。

整理

全局GC会选择evacuate/整理这些碎片化的内存区域。你可以把整理当做是在pc上的硬盘碎片化整理。我们把活动对象复制进当前没被整理的内存区域。通过这种方式，我们就能够充分利用非活动对象它们分散的内存区域了。

一种潜在的缺点就是当我们分配长时间运行的活动对象时，我们需要很高的代价去复制这些对象，这就是为什么我们选择仅仅整理高度碎片化的区域，且在其他的地方只执行清除，而并不会去复制这些对象。

分代布局

在v8中堆被分割进被称为代（generations）的区域.
下面是初代(进一步被分割成 ‘nursery’ and ‘intermediate’ sub-generations)和老生代。对象首先被分配进nursery。如果它们还活跃在下一个GC，它们会被送入初代的intermediate中。如果它们还活跃在另一个GC，它们将会被移到老生代中。

在垃圾收集器有这样一个术语：“多代假设”.大部分对象基本上在初代就会被清除。我们以GC的角度，换句话说，大部分对象它们在分配后，立即就会变成不可访问的。这种情况不仅仅是在v8和js语言上，它同样发生在大部分的动态语言上。

v8的多代堆布局就是为了揭露对象运行时的事实情况而设计的。GC本身就是一个整理/活动的GC，这就意味着它能复制活跃在垃圾收集器中的对象。这似乎不合常理：因为在GC运行时，复制对象是非常昂贵的。但是根据多代假设，我们所了解是仅仅只有一部分对象活跃在垃圾收集器中。仅仅是移动这些活跃的对象，所有其他的收集的对象都会变成隐式垃圾。这意味着我们仅仅只需要关注这些活跃对象，而不是所有的收集对象。

次要GC(Scavenger)

在v8中有两个垃圾收集器。来自于堆的主要垃圾收集器.来自于初生代的次要垃圾收集器主要GC在堆中起作用，但是多代假设告诉我们，新声明的对象非常可能需要被收集。

在次要GC中，活跃在初生代的被收集的对象总是被转移到一个新的区域。v8在初生代有一个名为’semi-space’的设计。它意味着为了满足转移的需要，在全部空间中，有过半的空间总是空着的。在整个清除期间最初的空的空间被称为 ‘To-Space’。那些我们复制对象所在的空间被称为 ‘From-Space’。有一个最糟糕的情况就是当每个对象活跃在清除阶段时，我们必须复制整个对象。
