传统与现代继承模式的比较

“传统”这个术语只是单词”class(类)”的一种表现形式。“现代”模式表述为：其他任何不需要以类的方式考虑的模式。

使用类式继承时的预期结果

实现类式继承的目标是通过构造函数Child()获取来自另外一个构造函数Parent()的属性，从而创建对象。

//父类
function Parent(name){
    this.name=name || "Nei";
};

// 添加功能
Parent.prototype.say=function(){
    return this.name;
};

//子类
function Child(){};

inherit(Child,Parent);

inherit函数非原生api，需要自行实现。现在我们来看看怎么实现它。

类式继承模式#1——默认模式

一种默认模式就是使用Parent构造函数创建一个对象，然后将该对象赋值给子类Child的原型。下面是inherit函数第一种实现：

function inherit(C,P){
    C.prototype=new P();
};

牢记一点，原型属性应该指向一个对象，而不是函数。所以它必须指向一个有父构造函数所创建的对象，而不是父构造函数本身。换句话说，要使用new操作符来创建的对象。

追溯原型链

使用这种模式时，同时继承了自身的属性（name），以及原型链上的属性和方法。那原型链是怎么样工作的呢？当new Parent()时，会创建一个块，保存这个对象，同时保存了name属性。但是当我们调用say方法时，say方法并不在这个块中，但是通过proto属性（指向Parent的prototype属性），我们找到了say方法。

当我们使用了inherit函数之后，又发生了什么呢？如图：

图中可以看到Child构造函数内部什么都没有，只有Parent实例的对象属性和方法被指向到Child的prototype上，Child的prototype上本身是什么都没有。当new Child()时，所创建的对象的proto指向Child类的prototype。所以当调用say方法时，在#3上并没有找到该方法，它会向#2上去寻找，在#2上也没有找到该方法，但是因为new Parent()的proto指向Parent的prototype，所以我们能在#1上找到该方法。

还有一个解释this指向的问题。如图：

var kid=new Child();
kid.name="De";
kid.say(); //输出 De


在kid对象上生成了name属性，调用say时，此时的this指向kid。

此模式缺点

在继承时，我们将自身属性和原型链上的属性和方法都继承了，但是有时并不需要这些自身属性。在实例化时，并不支持子构造函数传参。Code：

var s=new Child('Seth');
s.say(); // 输出 "Nei";

类式继承模式#2——借用构造函数

此模式解决了默认模式自构造函数不能传参的问题。通过一些方法借用父类构造函数，传递子对象以绑定到this，并且转发任意的参数。Code：

function Child(a,c,b){
    Parent.apply(this,arguments);
};

当时这种方法只能继承Parent自身的属性和方法，不能继承原型链上的属性和方法。

//父构造函数
function Article(){
    this.tags=['js','css'];
};
var article=new Article();

function BlogPost(){};
BlogPost.prototype=article; //默认模式继承
var blog=new BlogPost();

function StaticPage(){
    Article.call(this);
};
var page=new StaticPage();

console.log(article.hasOwnProperty('tags')); //true
console.log(blog.hasOwnProperty('tags')); //false
console.log(page.hasOwnProperty('tags')); //true

2返回了false，因为tags属性并没有在blog对象上，而是在原型链上，而page对象上的tags是借用了构造函数时，获取父类的副本，而不是引用。

当我们修改tags，看看差异：

blog.tags.push('html');
page.tags.push('php');
console.log(article.tags.join(','));输出 js,css,html

因为blog和article的tags指向了同一个数组，而page的targs时重新创建的的一个副本。

原型链

当使用本模式时，原型链时如何工作的呢？Code：

function Parent(name){
    this.name=name | "Adam";
};

Parent.prototype.say=function(){
    return this.name;
};

function Child(name){
    Parent.apply(this,arguments);
};

var kid=new Child('Nei');
kid.name; //输出 Nei
typeof kid.say //输出 undefined

最后为什么会输出undefined呢？因为借用继承只会继承父类的自身属性和方法。并没有继承到父类的prototype属性和方法。所以也就找不到say方法。

通过借用构造函数实现多重继承

借用多个构造函数实现多重继承。Code：

function Cat(){
    this.legs=5;
    this.say=function(){
        return 'miao'
    };
};

function Bird(){
    this.wings=3;
    this.fly=function(){
        return true;
    };
};

function Child(){
    Cat.apply(this);
    Bird.apply(this);
}

优缺点

缺点：

  • 无法继承父类原型链上的属性和方法，并且原型也仅是添加可重用方法以及属性的位置，它并不会为每个实例重新创建原型。
    优点：
  • 可以获得父类的自身成员的真实副本，也不会存在子对象意外覆盖父对象舒心的风险。

类式继承模式#3——借用和设置原型

顾名思义就是先借用父类构造函数，然后再设置原型。Code：

function Child(a,b,c,d){
    Parent.apply(this,arguments);
};
Child.prototype=new Parent();

此模式优点：

  • 继承了父类的自身成员和原型链上的成员。

缺点：

  • 自身成员被继承了两次。
  • constructor紊乱。子类构造函数应该指向本身，但是却指向了父类构造函数。

Orinoco垃圾回收器[译]

发表于 2019-01-30 |

在过去的许多年，v8的垃圾回收器发生了很大的变化。这次，Orinoco项目采用了一个连续，能够并行且并发和回退功能的异常强大的收集器。

垃圾收集器会周期性的执行下列几个基本的任务：

  • 鉴别正在运行或者已经没被调用的对象
  • 回收/再利用已经没被调用的对象的内存
  • 清理内存

这些任务一般按顺序执行，有时也会交错执行。在主线程上一些直截了当的方法暂停js执行和按顺序执行所有的任务，这样会造成不可避免和潜在的问题，我们在之前的博客谈论过.

主要GC

GC从堆中收集垃圾

全局GC发生在三个阶段：标记，清楚，整理

标记

垃圾收集器必不可少的就是找出哪些对象能被收集。收集器使用可达性(reachability)代理活跃度(liveness).这意味着在运行时当前一些可访问(reachable)的对象将保留，而那些不可访问对象将被收集起来。

标记就是可访问对象被找到的过程，GC就是在一系列一直被称为根集(root set)的对象指针开始的。这其中包括执行栈和全局对象。遵循一个指针对应一个对象，然后将这个对象标记为可访问的。GC持续递归的跟踪这些对象的指针，直到每个可访问对象被找到和标记。

清除

清除就是非活动对象被添加到空闲列表，所处的那块内存被清理的过程。一旦标记完成，GC会找到不可访问对象的那块连续内存，然后将它们添加到适当的空闲列表里。所谓的空闲列表就是由可以快速查找的内存块分出来的。在未来，当我们想要分配内存时，我们仅仅只需要考虑空闲列表，并且找到一块大小合适的内存块。

整理

全局GC会选择evacuate/整理这些碎片化的内存区域。你可以把整理当做是在pc上的硬盘碎片化整理。我们把活动对象复制进当前没被整理的内存区域。通过这种方式，我们就能够充分利用非活动对象它们分散的内存区域了。

一种潜在的缺点就是当我们分配长时间运行的活动对象时，我们需要很高的代价去复制这些对象，这就是为什么我们选择仅仅整理高度碎片化的区域，且在其他的地方只执行清除，而并不会去复制这些对象。

分代布局

在v8中堆被分割进被称为代（generations）的区域.
下面是初代(进一步被分割成 ‘nursery’ and ‘intermediate’ sub-generations)和老生代。对象首先被分配进nursery。如果它们还活跃在下一个GC，它们会被送入初代的intermediate中。如果它们还活跃在另一个GC，它们将会被移到老生代中。

在垃圾收集器有这样一个术语：“多代假设”.大部分对象基本上在初代就会被清除。我们以GC的角度，换句话说，大部分对象它们在分配后，立即就会变成不可访问的。这种情况不仅仅是在v8和js语言上，它同样发生在大部分的动态语言上。

v8的多代堆布局就是为了揭露对象运行时的事实情况而设计的。GC本身就是一个整理/活动的GC，这就意味着它能复制活跃在垃圾收集器中的对象。这似乎不合常理：因为在GC运行时，复制对象是非常昂贵的。但是根据多代假设，我们所了解是仅仅只有一部分对象活跃在垃圾收集器中。仅仅是移动这些活跃的对象，所有其他的收集的对象都会变成隐式垃圾。这意味着我们仅仅只需要关注这些活跃对象，而不是所有的收集对象。

次要GC(Scavenger)

在v8中有两个垃圾收集器。来自于堆的主要垃圾收集器.来自于初生代的次要垃圾收集器主要GC在堆中起作用，但是多代假设告诉我们，新声明的对象非常可能需要被收集。

在次要GC中，活跃在初生代的被收集的对象总是被转移到一个新的区域。v8在初生代有一个名为’semi-space’的设计。它意味着为了满足转移的需要，在全部空间中，有过半的空间总是空着的。在整个清除期间最初的空的空间被称为 ‘To-Space’。那些我们复制对象所在的空间被称为 ‘From-Space’。有一个最糟糕的情况就是当每个对象活跃在清除阶段时，我们必须复制整个对象。

tcp/ip分层管理

tcp/ip协议族分为四个层次，应用层 ⟶ 传输层 ⟶ 网络层 ⟶ 数据链路层,为什么需要分层呢，难道一层不好吗？还省得麻烦事，一层有一层，，每一层都需要不同的规范，有什么好处呢？

个人的理解，分层的原因在于，分层之后，某处的改动不会影响整体，将一个大的东西分成几个部分之后，设计来说相对于简单了，我们需要考虑的就是每个层次的设计，然后层次与层次之间的接口设计，而不要去设计一个巨大的整体。

应用层

应用层就是我们使用者打交道的这一层，http协议，ftp，dns服务也包括在其中。

传输层

传输层包括两种协议 tcp协议和udp协议。

网络层

网络层决定了数据包如何从一台计算机传输到另一台计算机

数据链路层

底层，硬件部分。包括操作系统，硬件设备，光驱，光纤等物理可见设备。

tcp/ip通信传输流

配图

某个用户在应用层发出请求，想查看某个网页时，传输层会把从应用层接收到的数据（http请求报文）进行分割，添加序号和端口号，然后再转发给网络层，网络层再会其上添加mac地址，这样数据就发送出去了。

发送端在层与层之间传输数据时，每经过一层，必定会添加改成的首部信息。接收端则相反，每经过一层，就是去掉相对应的首部信息。这种将数据包装起来的做法叫做封装.

与http关系密切的协议:ip、tcp和dns

ip协议

ip协议属于网络层，ip协议的作用就是负责把数据包传送给对方，但是需要满足很多条件，其中比较重要是这两个条件；1 ip地址，2 mac地址。其中ip地址就是节点所分配到的地址，mac地址指的是网卡所属的固定地址，ip地址可以变换，但是mac地址一般不会更改。

我们知道数据包传输时，需要进过许多的中转站，如果中转的时候我们只有下一站的ip地址，我们可以通过ARP协议拿到下一站的mac地址，ARP协议可以通过ip地址反查mac地址。

tcp协议

tcp协议属于传输层，它提供了字节流服务，字节流服务就是将大块数据分割成以报文段为单位的数据包。并且还是稳定可靠的。为了稳定可靠的传输，tcp协议采用了三次握手策略，即发送端将带有SYN的数据包发送给接收端，接收端接收到，再将带有SYN/ACK的数据包返回给发送端，发送端接收到，再将带有ACK的数据发送给接收端，整个过程结束。如果其中某一个过程终端，tcp协议会再次以相同顺序发送相同的数据包。

dns服务

dns服务提供ip到域名的解析服务,我们一般用域名来访问主机,但是各种各样的域名对于计算机来说,全部识别起来是相当困难的,计算机还是喜欢ip地址这种纯数组组合,所以诞生了dns服务,它提供了ip到域名的映射,使得我们输入域名时,解析域名,找到该域名对应的ip。

ECMA规范之Array.prototype.push

发表于 2019-01-25 | 分类于 ECMA |

我们几乎每天都要用到push方法，我们真的明白在调用push方法后，浏览是怎么运行这个方法？

其中ecma规范第三版(1999)是这样定义的：

  • 1 调用对象的[Get]方法获取参数列表中第一个参数长度
  • 2 将结果用ToUint32方法赋值给n
  • 3 在参数列表中获取下一个参数，如果没有，直接跳到第七步
  • 4 调用对象的[Put]方法将当前参数Tostring，put到当前对象中去
  • 5 n自增1
  • 6 回到第三步，循环
  • 7 调用对象的[Put]方法，将参数长度和n put 到当前对象中去
  • 8 返回n

push原生方法的长度属性为1

到了2011年，我们push规范进行了变更，大致变更如下：

  • 1 调用ToObject处理this对象，将结果赋值给O
  • 2 调用O对象的内部方法[Get]将第一个参数长度赋值给lenVal
  • 3 将ToUint32(lenVal)赋值给n
  • 4 将传递给函数调用的参数从左到右，依次赋值给items
  • 5 如果items不为空，重复下列方法：
    a.移除items中的第一个元素，并且将移除元素的值赋给E
    b.调用O对象的[Put]内部方法，在O对象上设置ToString(n)，E和true属性。
    c.n自增1
  • 6 当items为空时，调用O对象的[Put]内部方法，设置O对象的参数长度，n和true等属性。
  • 7 返回n

NOTE：
push方法被刻意设计为一种通用的方法；它并不要求this对象是一个数组对象。因此它能使用特定方法被其他对象调用。

var obj={

    addElem : function addElem(elem){

        [].push.call(this,elem);
        //Array.prototype.push.call(this,elem);
    }
}

obj.addElem('1');
obj.addElem('2');
console.log(obj.length) //result is 2

时光回到2015年，ecma规范又一次更新，push当然也是更新了

我们来看看具体有哪些更新：

  • 1 ToObject(this)并且将值赋给O
  • 2 调用ReturnIfAbrupt(O) ps：这个函数是啥意思？目前还没弄懂，弄懂再回来补充
  • 3 将O的长度调用ToLength()赋值给len ps：这个toLength又是什么呢？
  • 4 调用ReturnIfAbrupt(len)
  • 5 步骤还是跟第五版一样，将所有传递进函数的参数从左至右赋值给items。
  • 6 将items中的元素个数赋值给argCount
  • 7 如果len + argCount > 2的53次方减一，抛出TypeError exception
  • 8 如果items不为空，重复下列行为：
    a.从items移除第一个元素，并且将这个元素的值赋给E
    b.调用Set方法设置O对象的ToString(len)，E和true属性
    c.ReturnIfAbrupt(setStatus)
    d.len自增1
  • 9 如果items为空，调用Set方法，设置O对象”length”，len和true属性
  • 10 ReturnIfAbrupt(setStatus).
  • 11 返回len

es6内部抽象程度更高，更加规范，调用了很多方法来处理不同的过程，那让我们来了解研究这些方法到底是干什么的。

命名空间模式

命名空间有助于减少程序中所需要的全局变量的数量，并且同事还有助于避免命名冲突或过长的名字前缀。我们可以创建一个全局对象，将所需的方法，属性添加到这个对象中。避免了变量全局污染。而且同样能避免与第三库的命名冲突。但是同样存在一些缺点：

  • 使用的时候，需要附加全局对象这个前缀，增加了很多重复代码。
  • 长嵌套名字意味着更长的属性解析查询时间
  • 一个全局对象意味着，任何部分的代码都能修改这个全局对象，并且影响所有调用该对象的地方。

通用命名空间函数

当我们往这个全局对象添加一个属性时，可能覆盖之前已经定义的属性了，所有我们在添加时，先得检测这个添加的属性在这个全局对象中是否已经存在。Code：

var MYAPP={};

if(typeof MYAPP === "undefined"){
    var MYAPP={};
}

var MYAPP = MYAPP || {};

这种检测一个对象好使，但是检测链式对象时，显得乏力，代码也会变得臃肿。所有我们需要设计一个可以检测链式对象的方法。Code：

MYAPP.namespace('MYAPP.modules.module2');

具体的方法实现。Code：

var MYAPP = MYAPP || {};

MYAPP.namespace = function (ns_string){
    var parts = ns_string.split('.'),
        parent = MYAPP,
        i;
    if(parent[0] === "MYAPP"){
        parts = parts.slice(1);
    }

    for(i=0; i < parts.length; i++>){
        if(typeof parts[i] === "undefined"){
            parent[parts[i]]={};
        }
        parent = parent[parts[i]]
    }
    return parent;
};

声明依赖关系

在函数或顶部声明代码所依赖的模块是一个很好的方法。因为这种声明仅涉及创建一个局部变量并使其指向所需的模块。Code：

var myFunction = function (){
    var event = YAHOO.util.Event,
        dom = YAHOO.util.Dom;
};

这种模式有如下几个优点：

  • 显示的依赖声明表明需要的特定脚本已经包含在该页面中。
  • 在函数顶部的前期声明可以使页面维护者很容易地发现并解析依赖。
  • 解析局部变量的速度总是要比解析全局变量（YAHOO）要快，甚至比使用全局变量的嵌套属性（YAHOO.util.Dom）还要快。
  • 将第三方模块赋值给重命名的局部变量，可以拥有更好的代码量，因为我们可以使用更短的局部变量。

私有属性和方法

js中并没有特殊的语法来表示私有、保护、或公共属性和方法。因为js中所有对象的成员都是公共的。

比如我们随便创建一个对象：

var obj{
    name:'nei',
    run:function(){
        console.log('Run');
    }
};

obj.name //公共的，可以随便访问的。

即使使用构造函数同样如此：

function Obj(){
    this.name='Nei';
    this.run=function(){
        console.log('run');
    };
}

var o1=new Obj();

o1.name //公共的 可以随便访问的

但是有时我们需要私有成员，所以诞生了下面的这些生成私有成员的方法。

私有成员

虽然js本身没有直接创建私有成员的方法，但是我们将变量控制在函数作用域内，保证外部访问不到它，这样一样可以创建私有成员，比如我们创建一个类，在类里面创建一个局部变量，然后用公共方法将它暴露出去，这样我们就可以创建一个私有变量，但是有个公共方法又可以访问到它。

function Nei(){
    var _name='nei';
    this.getName=function(){
        return _name;
    };
}

var n=new Nei();
n.name   //undefined
n.getName() //"nei"

模块模式

模块模式就如下几种的集合：

  • 命名空间
  • 即时函数
  • 私有和特权成员
  • 声明依赖

沙箱模式

顾名思义就是提供了一个可用于模块运行的环境，且不会对其他模块和个人沙箱造成任何影响。

全局构造函数

命名空间模式中有一个全局对象，在沙箱模式中，我们可以设置一个全局构造函数，创建一个隔离的沙箱环境。Code：

new Sandbox(function(box)){
    ....
});

通过添加新特征，使这个模式更先进：

  • 通过第三章的强制new模式，可以假设在创建对象是不需要new操作符。
  • 构造函数可以接受一个额外的配置参数（或多个），其中该参数指定了对象实例所需要的模块名。希望代码是模块化的，因此绝大部分构造函数提供的功能将被限制在模块中。

静态成员

静态成员分为公有静态成员和私有静态成员

公有静态成员

所谓公有就是直接暴露在对象的属性上，外部可以访问。Code：

var Garget=function(){};

//静态方法
Garget.isShiny=function(){
    return "you bet";
};

//普通方法
Garget.prototype.setPrice=function(price){
    this.price=price;
};

//可以直接调用静态方法
Garget.isShiny();//输出 you bet

//普通方法必须创建一个实例，实例调用
var demo1=new Garget();
demo1.setPrice(12);

试图以静态方法调用一个实例方法是无法正常运行的。同样，如果使用实例调用静态方法也是无法正常运行的：

typeof Garget.setPrice; // undefined
typeof demo1.isShiny; // undefined

如果静态方法也能够被实例调用，只需重新写一个原型方法即可：

Garget.prototype.isShiny=Garget.isShiny;
demo1.isShiny; // 输出 “you bet”

但是要注意这里的this指向问题，执行Garget.isShiny时，this指向Garget。执行demo1.isShiny时，this指向demo1。

我们可以用instanceof确定方法是如何被调用的：

var Garget=function(price){
    this.price=price
};

Garget.isShiny=function(){
    var msg="you bet";

    is(this instanceof Garget){
        msg += ", it costs $"+ this.price + "!";
    }
    return msg;
};

Garget.prototype.isShiny=function(){
    return Garget.isShiny.call(this);
};

//调用静态方法
Garget.isShiny; // 输出 "you bet"

//实例调用
var d1=new Garget(12);
d1.isShiny(); // 输出 "you bet,it costs $12!"

私有静态成员

私有静态成员具有如下属性：

  • 以同一个函数所创建的所有对象共享该成员
  • 构造函数外部不可访问。

利用立即执行函数实现一个简单的私有静态成员：

var Garget=(function(){
    var counter=0;
    return function(){
        console.log(counter+=1);
    };
})();

//多次调用
var d1=new Garget(); // 1
var d2=new Garget(); // 2
var d3=new Garget(); // 3

现在，我们通过一个特权方法将私有属性暴露出去：

var Garget=(function(){
    var counter=0,
        NewGarget;

    NewGarget=function(){
        counter+=1;
    };

    //私有特权方法
    NewGarget.prototype.getLastId=function(){
        return counter;
    };

    //重写构造函数
    return NewGarget;
})();

//测试
var d1=new Garget();
d1.getLastId(); // 1
var d2=new Garget();
d2.getLastId(); // 2
var d3=new Garget();
d3.getLastId(); // 3

静态属性可以包含非实例相关的方法和数据，并且不会为每个实例重新创建静态属性。
